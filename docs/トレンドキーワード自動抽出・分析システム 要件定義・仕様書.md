## 1. プロジェクト概要・目的

X（Twitter）における突発的なバズ（広さ）と、noteにおけるコンテンツ需要（深さ）を自動で収集・突合し、「ユーザーの熱量が著しく高く、かつコンテンツ化に値する強いキーワード」をデータ駆動で発掘するシステムを構築する。 直感や手動リサーチに頼らず、再現性のあるコンテンツ企画・マーケティングの基盤とすることを目的とする。

## 2. システムアーキテクチャ概要

本システムは以下の4つのフェーズで構成されるデータパイプラインである。

1. **Collection (収集):** Yahoo!リアルタイム検索およびnoteから定期的にデータを取得。
    
2. **Filtering (選別):** ルールベースおよびローカルLLMを用いてノイズを除外。
    
3. **Cross-Reference (突合・解析):** 異なるプラットフォームのデータをキーワードを軸に結合し、スコアリング。
    
4. **Visualization (可視化):** ダッシュボード等で一覧化・比較分析。
    

## 3. 機能要件

### 3.1. データ収集機能 (Data Collection)

- **Xトレンドデータ取得 (Yahoo!リアルタイム検索)**
    
    - 手法: Playwright + Stealth Pluginを用いたスクレイピング。
        
    - 対象: トレンドランキング上位ワード、および特定シードワード（例:「ツール」）のサジェスト。
        
    - 取得項目: キーワード名、24時間ツイート数、感情分析比率（ポジ/ネガ）、取得日時。
        
- **noteランキングデータ取得**
    
    - 手法: 非公式APIエンドポイントへの直接HTTPリクエスト（推奨）、またはスクレイピング。
        
    - 対象: 特定カテゴリ（テクノロジー、ビジネス等）のランキング上位記事。
        
    - 取得項目: 記事タイトル、記事URL、設定ハッシュタグ、スキ数、公開日時。
        

### 3.2. フィルタリング機能 (Filtering)

- **ルールベース・フィルタリング (1次選別)**
    
    - 事前定義したNGワード（政治、芸能、災害、アダルト等）との完全一致または部分一致による除外。
        
    - ツイート数が設定した最低閾値（例: 500件）を下回るワードの除外。
        
- **LLMベース・フィルタリング (2次選別)**
    
    - 手法: ローカルLLM（Ollama経由）へのプロンプト投入による文脈判定。
        
    - 判定基準: 抽出されたキーワードが、ターゲットとするドメイン（技術、生産性、Web開発など）に関連するかどうかを判定（True/False、または1〜5のスコア化）。
        

### 3.3. 突合・解析機能 (Cross-Reference & Scoring)

- Xから抽出された「有効キーワードリスト」と、noteの「上位記事データ（タイトル・タグ）」をマッチングさせる。
    
- **スコアリングロジック（案）:**
    
    - `X_Buzz_Score` = (24時間ツイート数) × (ポジティブ感情比率)
        
    - `note_Demand_Score` = 対象キーワードを含むランキング記事の総スキ数（または出現頻度）
        
    - `Total_Trend_Score` = (X_Buzz_Scoreの正規化値) × (note_Demand_Scoreの正規化値)
        

### 3.4. 可視化・通知機能 (Visualization)

- **一覧ダッシュボード:**
    
    - 抽出・スコアリングされたキーワードをテーブル形式で表示。
        
    - 各スコアでのソート機能。
        
- **マトリクス分類:**
    
    - 縦軸（note需要）、横軸（Xバズ）による4象限プロット。
        
        - 第1象限: X高・note高（超お宝・即執筆ワード）
            
        - 第2象限: X低・note高（ロングテール・ストックワード）
            
        - 第3象限: X低・note低（対象外）
            
        - 第4象限: X高・note低（先行者利益・トレンドワード）
            

## 4. 非機能要件

- **実行頻度:** 1日1〜2回（深夜バッチ・日中バッチ）の定期実行。
    
- **Bot対策回避:** Yahoo!リアルタイム検索へのアクセスは、リクエスト間隔のランダム化、User-Agent偽装、Stealth Pluginの適用を必須とする。APIへの過度な負荷を避ける。
    
- **ランニングコスト:** LLM判定をローカル環境（Ollama）で行うことで、トークン課金をゼロに抑える。
    

## 5. データベース設計（概念モデル）

### Table: `x_trends` (X/Yahooトレンド履歴)

- `id` (PK)
    
- `keyword` (String)
    
- `tweet_count_24h` (Int)
    
- `positive_ratio` (Float)
    
- `is_filtered_out` (Boolean) - ノイズ判定フラグ
    
- `scraped_at` (Timestamp)
    

### Table: `note_ranking` (note上位記事データ)

- `id` (PK)
    
- `article_id` (String) - note側の記事ID
    
- `title` (String)
    
- `tags` (Array of Strings)
    
- `like_count` (Int)
    
- `category` (String)
    
- `scraped_at` (Timestamp)
    

### Table: `keyword_analysis` (突合・スコア結果)

- `id` (PK)
    
- `date` (Date) - 分析対象日
    
- `keyword` (String)
    
- `x_score` (Float)
    
- `note_score` (Float)
    
- `total_score` (Float)
    
- `status` (String) - 象限等の分類ステータス
    

## 6. 今後の拡張性（オプション）

- Google Trends API (`google-trends-api`) の統合による、中長期的な検索ボリューム推移の掛け合わせ。
    
- 抽出した「お宝キーワード」をベースにした、記事構成案（アウトライン）のLLM自動生成機能の追加。
