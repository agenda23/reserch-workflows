## 1. プロジェクト概要・目的

X（Twitter）における突発的なバズ（広さ）と、noteにおけるコンテンツ需要（深さ）を自動で収集・突合し、「ユーザーの熱量が著しく高く、かつコンテンツ化に値する強いキーワード」をデータ駆動で発掘するシステムを構築する。 直感や手動リサーチに頼らず、再現性のあるコンテンツ企画・マーケティングの基盤とすることを目的とする。

## 2. システムアーキテクチャ概要

本システムは、noteプラットフォームの蓄積実需（ストック需要）を起点とし、X（Twitter）でのリアルタイムの話題性（瞬間風速）を後から突合する「note先行・X話題性突合」アプローチを採用する。これにより、超短期で消え去るXバズのノイズを完全に遮断し、中長期的な技術コンテンツ需要を持続的に調査可能とする。
システムは以下の4つのフェーズで構成される。

1. **Collection (収集):** noteカテゴリ別人気記事およびnoteハッシュタグ統計を先行取得し、シードサジェストとマージして「実需キーワード候補」を抽出。
    
2. **Filtering (選別):** 強化されたルールベースNGワードおよび多言語NLP埋め込みモデル（`intfloat/multilingual-e5-small`）を用いて、ターゲットテーマとの「マージン付き相対コサイン類似度」を判定しノイズを厳格に除外。
    
3. **Cross-Reference (突合・解析):** フィルタリングを通過したキーワードに対して個別にXのポスト件数・感情比率を突合スクレイピングし、noteのスキ数・ハッシュタグ投稿件数と掛け合わせてスコアリング。
    
4. **Visualization (可視化):** 4象限マトリクス（Recharts）、および新設の「ハッシュタグ調査」タブを用いた需要ボリューム比較分析。
    

## 3. 機能要件

### 3.1. データ収集・突合機能 (Data Collection & Cross-Reference)

- **noteランキングデータ取得 (先行収集フェーズ1)**
    
    - 手法: 非公式APIエンドポイントへの直接HTTPリクエスト。
        
    - 対象: 特定カテゴリ（テクノロジー等）の人気記事。
        
    - 取得項目: 記事タイトル、記事ID、設定ハッシュタグ、スキ数、カテゴリ。
        
- **noteハッシュタグ統計情報取得 (先行収集フェーズ2)**
    
    - 手法: noteハッシュタグ統計API (`GET /api/v2/hashtags/{hashtag}`) へのリクエスト。
        
    - 対象: シードキーワード（例:「AI」, 「ツール」, 「効率化」, 「開発」）ごとの統計。
        
    - 取得項目: ハッシュタグ名、累積投稿件数、関連ハッシュタグリスト。

- **Qiita / Zenn トレンド取得 (先行収集フェーズ3)**

    - Qiita: トップページからトレンドタグをPlaywrightでスクレイピング。
    - Zenn: RSSフィード (`https://zenn.dev/feed`) から最新記事タイトルをパース。
    - 取得項目: トレンドタグ名・記事タイトル（クレンジング後のキーワード候補）。

- **Brain / Tips 有料コンテンツランキング取得 (先行収集フェーズ4)**

    - 手法: `playwright-stealth` によるフィンガープリント偽装を適用したPlaywrightスクレイピング。
    - 対象: Brain・Tipsのランキング上位コンテンツのタイトル/タグ情報。
    - 効果: 「無料の技術トレンド」と「有料コンテンツ市場の金銭的実需トレンド」を同時に捕捉。

- **X話題性突合スクレイピング (後続突合フェーズ)**
    
    - 手法: `playwright-stealth` を適用したPlaywrightによる個別キーワードのYahoo!リアルタイム検索結果ページのスクレイピング。
        
    - 対象: フィルタリングを通過した「実需キーワード候補」（最大40件）。
        
    - 取得項目: キーワード名、Xポスト数、ポジティブ感情比率。

- **供給量（競合コンテンツ数）の取得 (分析補助)**

    - Qiita API (`GET /api/v2/tags/{tag}`) からタグ別記事数を取得。
    - noteハッシュタグキャッシュデータ（`note_hashtags` テーブル）からの補足加算。
    - 用途: ブルーオーシャン度（Priority Score）の算出に使用。
        

### 3.2. フィルタリング機能 (Filtering)

- **ルールベース・フィルタリング (1次選別)**
    
    - 定義されたNGワード（ツールド、開発行為、修学旅行など、一般・固有名詞ノイズ）との部分一致による除外。
        
- **多言語NLP類似度フィルタリング (2次選別)**
    
    - 手法: ローカルCPU上で軽量な多言語埋め込みモデル `intfloat/multilingual-e5-small` を実行。
        
    - 判定ロジック: 
        1. ターゲットテーマ（IT技術、プログラミング、AI等）に対する最大類似度 `Target_Score` を算出。
        2. ネガティブテーマ（スポーツ、芸能、ニュース、行政手続き等）に対する最大類似度 `Negative_Score` を算出。
        3. 判定式: `(Target_Score - Negative_Score >= 0.02)` かつ `Target_Score >= 0.81` のマージン付き相対比較を満たすもののみを採用。これにより、形態素の部分一致によるノイズ（「ツールドフランス」「開発行為」等）を完璧に遮断する。
        

### 3.3. 突合・解析機能 (Cross-Reference & Scoring)

- 抽出された「実需キーワード」に対して、X話題性突合スクレイピングで得られたツイート数と、note側のスキ数・ハッシュタグ投稿件数を掛け合わせて正規化スコアを算出。
    
- **スコアリングロジック:**
    
    - `X_Buzz_Score` = (X突合でのポスト数) × (ポジティブ感情比率) ※サジェストワード等の初期バズ値は100として突合。
        
    - `note_Demand_Score` = (E5類似度0.78以上のnote記事スキ数 × 類似度自乗重み付け) + (関連するシードハッシュタグ投稿件数の対数スケールボーナス `log10(post_count) * 2`)
        
    - `Normalized_Score` = 最小値を `0.05` にクリップして正規化したスコア（0へばりつきを防止し、グラフ上にバランスよく配置するため）。
        
    - `Total_Trend_Score` = (X_Buzz_Scoreの正規化値) × (note_Demand_Scoreの正規化値)
        

### 3.4. 可視化・通知機能 (Visualization)

- **一覧ダッシュボード:**
    
    - 抽出・スコアリングされたキーワードをテーブル形式で表示。
        
    - 各スコアでのソート機能。
        
- **マトリクス分類:**
    
    - 縦軸（note需要）、横軸（Xバズ）による4象限プロット。
        
        - 第1象限: X高・note高（超お宝・即執筆ワード）
            
        - 第2象限: X低・note高（ロングテール・ストックワード）
            
        - 第3象限: X低・note低（対象外）
            
        - 第4象限: X高・note低（先行者利益・トレンドワード）
            

## 4. 非機能要件

- **実行頻度:** 1日1〜2回（深夜バッチ・日中バッチ）の定期実行。
    
- **Bot対策回避:** Yahoo!リアルタイム検索・Brain・Tipsへのアクセスは、`playwright-stealth` によるヘッドレスブラウザシグネチャ偽装（`navigator.webdriver` 等）、リクエスト間隔のランダム化（ミリ秒単位のゆらぎ）、User-Agent偽装、画面解像度ランダム化を必須とする。APIへの過度な負荷を避ける。
    
- **ランニングコスト:** NLP類似度判定をローカル環境で動作する軽量モデル（Sentence Transformers）で行うことで、トークン課金をゼロに抑える。
    

## 5. データベース設計（概念モデル）

### Table: `x_trends` (X/Yahooトレンド履歴)

- `id` (PK)
    
- `keyword` (String)
    
- `tweet_count_24h` (Int)
    
- `positive_ratio` (Float)
    
- `is_filtered_out` (Boolean) - ノイズ判定フラグ
    
- `scraped_at` (Timestamp)
    

### Table: `note_ranking` (note上位記事データ)

- `id` (PK)
    
- `article_id` (String) - note側の記事ID
    
- `title` (String)
    
- `tags` (Array of Strings)
    
- `like_count` (Int)
    
- `category` (String)
    
- `scraped_at` (Timestamp)
    

### Table: `keyword_analysis` (突合・スコア結果)

- `id` (PK)
    
- `date` (Date) - 分析対象日
    
- `keyword` (String)
    
- `x_score` (Float)
    
- `note_score` (Float)
    
- `total_score` (Float)
    
- `status` (String) - 象限等の分類ステータス

- `supply_count` (Integer) - 既存の競合コンテンツ総数（Qiita + note投稿件数の合算）

- `priority_score` (Float) - ブルーオーシャン度（Demand Score / log10(supply_count + 2)）
    

### Table: `note_hashtags` (noteハッシュタグ統計情報)

- `id` (PK)
    
- `hashtag` (String) - ハッシュタグ名（ユニーク）
    
- `post_count` (Int) - 累積投稿件数
    
- `related_tags` (Array of Strings) - 関連ハッシュタグ
    
- `updated_at` (Timestamp)
    

## 6. 今後の拡張性（オプション）

- **LLMによる動的な文脈精査判定の導入**: 
  軽量な多言語NLPモデル（Sentence Transformers）で9割の日常ノイズを高速に足切り（1次選別）した後、残った有望キーワードに対してのみ、ローカルLLM（Ollama等）または外部APIを介して「IT/テック系開発者にとって有益なトレンドか」をプロンプトで最終判定（2次選別）する二段階ハイブリッドフィルタリングの導入。
    
- Google Trends API (`google-trends-api`) の統合による、中長期的な検索ボリューム推移の掛け合わせ。
    
- 抽出した「お宝キーワード」をベースにした、記事構成案（アウトライン）のLLM自動生成機能の追加。

- **Zennタグの供給数取得**: 現在はQiita APIのみを使って競合数を算出しているが、Zenn APIでのタグ別記事数も加算することでより精度の高い供給量推定が可能。
