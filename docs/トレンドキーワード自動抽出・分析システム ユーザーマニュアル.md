# トレンドキーワード自動抽出・分析システム ユーザーマニュアル

本システムは、note（techカテゴリ人気記事やシードハッシュタグ）の深層需要を起点とし、X（Yahoo!リアルタイムトレンド検索）の話題性を後から突合することで、ローカルNLP（意味分析）を用いて「実需があり、かつ今注目されているキーワード」を自動抽出・4象限分析するシステムです。

## 1. システム概要

### 課題解決の背景とアプローチ
情報発信者や技術リサーチャーにとって、「何について解説した記事が実際に読まれているか（note）」「今どのような技術キーワードがバズっているか（X）」を両面から把握することは極めて重要です。しかし、X発祥のトレンドワードは超短期（数時間〜数日）で消え去るノイズが多く、noteの実需と同期しにくい課題があります。

そこで本システムは、**「note先行・X話題性突合」** アプローチを採用しています。
1. note人気記事・ハッシュタグ統計に加え、**Qiita/Zennトレンド**・**Brain/Tipsランキング**を先行取得し、「無料技術トレンド」と「有料コンテンツ市場の実需」を同時に捕捉して最大40件の候補キーワードを抽出します。
2. その後、抽出された候補キーワードに対して、個別にX上の話題性（ポスト数・感情比率）をYahoo!リアルタイム検索からスクレイピングして突合します。
3. **ローカルNLP（Sentence Transformers）** を利用して、「ターゲットテーマ」と「ネガティブテーマ（スポーツ・日常など）」の意味類似度を相対比較（マージン付き相対判定）し、さらに**当日の時事ニュースRSS**から自動生成した動的ネガティブテーマも加味して、技術と無関係な日常ノイズワードを遮断した上で、以下の4象限に自動分類します。

*   **第1象限（スター候補）**: noteでの需要（スキ数や投稿数）が高く、かつXでも話題になっている、今すぐ取り組むべき強力なトレンド。
*   **第2象限（安定需要）**: Xでの爆発力はないが、noteでは安定して読まれている（スキ数が多い）実需系キーワード。
*   **第3象限（対象外）**: Xでの話題性もnote需要も低く、かつ技術的文脈から遠い（類似度判定が低い）キーワード。
*   **第4象限（急上昇トレンド）**: Xで話題になり始めているが、まだnoteでのコンテンツ蓄積が少ない、先行者利益を狙えるキーワード。

## 2. 動作環境・セットアップ

本システムは、DockerおよびDocker Compose環境さえあれば、ローカルPCのCPUのみ（有料APIキー不要）で即座に動作します。

### 前提条件
*   **Docker / Docker Desktop**: インストール済みであること
*   **Docker Compose**: 使用可能であること

### 起動手順
1.  プロジェクトのルートディレクトリに移動します。
2.  以下のコマンドを実行してコンテナをビルド・起動します。
    ```bash
    docker compose up -d --build
    ```
    *初回起動時は、NLPモデル (`intfloat/multilingual-e5-small`) の自動ダウンロードやPlaywrightのブラウザダウンロードが走るため、数分程度かかります。*

3.  起動後、ブラウザで以下のURLを開きます。
    *   **Web UI (フロントエンド)**: `http://localhost:5173/`
    *   **APIドキュメント (FastAPIバックエンド)**: `http://localhost:8000/docs`

### 停止手順
動作を終了する場合は、以下のコマンドを実行します。
```bash
docker compose down
```

## 3. 画面構成と基本操作

### ダッシュボード (Dashboard)

ダッシュボードは、収集したキーワードの分析結果をグラフィカルに可視化し、詳細を追うためのメイン画面です。

#### 1. データの新規収集実行
*   **「データ収集・分析を実行」ボタン**:
    クリックすると、非同期でYahoo!リアルタイムトレンドのスクレイピングと、note人気記事のAPI収集が開始されます。
    実行中は進捗状況（プログレスバー）が表示され、処理が完了すると自動でダッシュボードが更新されます。

#### 2. ブルーオーシャン推奨ランキング
*   収集データの中から、需要が高く競合（既存記事数）が少ないキーワードを **Priority Score** 順に最大5件表示します。
*   第1象限（超お宝）または第4象限（急上昇トレンド）に分類されたキーワードが対象です。
*   各エントリをクリックすると、下部テーブルの該当行へスクロールします。

#### 3. 4象限マトリクスグラフ (Recharts)
*   収集・突合されたキーワードが、**横軸（Xバズスコア: 話題の広さ）** と **縦軸（note需要スコア: 関心の深さ）** の平面上にプロットされます。
*   プロットされた点にマウスカーソルを合わせる（ホバーする）と、キーワード名、詳細なスコア、分類（象限）がツールチップで表示されます。
*   グラフ上の点をクリックすると、下部の「キーワード詳細テーブル」の該当行へスクロールし、ハイライト表示されます。

#### 4. キーワード詳細テーブル
*   収集されたキーワードの一覧をテーブル形式で表示します。
*   検索バーによるキーワード絞り込みや、各カラム（Xスコア、noteスコア、**供給数**、**優先度スコア**、分類ステータスなど）でのソートが可能です。
*   **供給数**: Qiita・noteの既存コンテンツ数の合算（競合の多さを示す）。数値が小さいほどブルーオーシャン。
*   **優先度スコア**: `Demand Score ÷ log10(供給数 + 2)` で算出するブルーオーシャン度。数値が高いほど「今すぐ書く価値が高い」。

### ハッシュタグ調査 (Hashtag Research)

ハッシュタグ調査画面は、システム設定の「シードキーワード」に指定されたハッシュタグについて、noteプラットフォーム全体の累積投稿ボリュームや関連ハッシュタグを一覧・ビジュアル分析するための画面です。

#### 1. 概要カード
*   **監視中ハッシュタグ総数**: 現在登録されているシードキーワードの数。
*   **最大投稿ボリューム**: 登録されているハッシュタグの中で、最も投稿件数が多いタグとその件数。
*   **総投稿ボリューム（市場規模）**: すべてのシードハッシュタグの累積投稿件数の合計。

#### 2. note ハッシュタグ需要調査テーブル
*   取得したハッシュタグ名、noteの累積投稿件数、**Qiita記事数**、およびnote APIから取得した関連ハッシュタグ（サジェストタグ）を一覧表示します。
*   **Qiita件数**: Qiita APIから取得した同名タグの記事数。noteとQiitaを横断して需要規模を把握できます。
*   関連ハッシュタグは、note内でそのキーワードと同時に使われやすいタグを示しており、コンテンツ企画の周辺テーマの探索に役立ちます。

#### 3. ボリューム比率（クロスメディア可視化）
*   **noteとQiita** それぞれの投稿件数を最大値基準（100%）で比較するグラデーション棒グラフを表示します。
*   例: noteでは活発だがQiitaでは少ないキーワードは「noteユーザー特有の実需」、両プラットフォームで多いキーワードは「テック全般で強い需要」と読み解けます。

### システム設定 (Settings)

システム設定画面では、NLPによるキーワード抽出の精度向上や、収集対象のカスタマイズを行えます。設定は保存され、次回の収集実行時から即座に適用されます。

#### 1. 基本設定
*   **ターゲットテーマ**:
    NLPの類似度判定の基準となる文章を入力します。
    *例: 「Web開発、React、Next.js、TypeScript、フロントエンド、バックエンド技術に関する話題」*
*   **類似度しきい値 (Similarity Threshold)**:
    ターゲットテーマとの類似度がこの値（デフォルト: `0.81`）未満のキーワードは「対象外」として除外され、4象限グラフ上にもプロットされません。
*   **NGワードリスト**:
    除外したいキーワードをカンマ区切りで入力します。ここに登録された語を含むトレンドは、類似度にかかわらず無条件で除外されます。
    *例: 「地震, 運行情報, 遅延, 事故」*
*   **note収集カテゴリ**:
    note APIから取得するカテゴリ（デフォルト: `tech`）を指定します。

#### 2. 類似度テスト（シミュレーター）
*   現在の「ターゲットテーマ」と「類似度しきい値」の設定で、特定のキーワードがシステムを通過（採用）するかどうかを事前テストできます。
*   テストしたいキーワードを入力して「テスト実行」をクリックすると、E5モデルを用いた類似度スコアと、採用判定結果（True / False）がリアルタイムに表示されます。

## 4. 機能と運用の詳細

### ターゲットテーマ（キーワード）の設定のコツ
本システムが採用している `intfloat/multilingual-e5-small` は文脈の意味を比較する高精度な多言語埋め込みモデルです。
単なる単語（例: `React`）ではなく、**「何に関するキーワードを抽出したいか」という具体的なコンテキスト（説明文）**をターゲットテーマに指定することが重要です。

*   **効果的な設定例**:
    > `ソフトウェア開発、AI技術、プログラミング、インフラ、Webサービス、SaaS開発に関する技術的なトレンドや話題`
*   **あまり効果的でない設定例**:
    > `技術, 開発, プログラミング` (単語のみだと判定が曖昧になり、類似度スコアのブレが大きくなります)

### 類似度しきい値のチューニングと相対判定ロジック
E5モデルは類似度スコアが比較的高く出やすい（無関係な日常語でも0.75〜0.78程度になる）性質があります。
本システムでは、この課題に対処するため、指定された「ターゲットテーマ」との類似度に加え、「日常ニュース、スポーツ、エンタメ」等の「ネガティブテーマ」との類似度も同時に計算する**相対比較ロジック**を導入しています。

具体的には、以下の条件を満たしたキーワードのみが判定を通過します。
1. **ターゲットテーマの類似度が、設定した「類似度しきい値」以上であること**
2. **（ターゲットテーマの類似度 - ネガティブテーマの類似度）が `0.02` 以上であること**（マージン判定）

これによって、「ツールドフランス（フランスの地名やスポーツが含まれ、開発行為と類似度が高めに出やすい）」のような、一部の単語マッチングによる誤判定をシャープに遮断できます。

*   **おすすめのしきい値**: **`0.81`** (デフォルト)
*   **厳しめに絞り込みたい場合**: **`0.82`〜`0.84`** に設定すると、技術的な専門用語や直球のITトピックのみが厳格に残ります。
*   **広く収集したい場合**: **`0.79`〜`0.80`** に設定すると、技術に隣接するビジネス、ガジェット、副業ノウハウなども柔軟に拾うことができます。

*設定の際は、まず「類似度テスト（シミュレーター）」機能でいくつかのテストワード（例: 「Python」「ツールドフランス」「大雨」「自動運転」）を入力し、狙い通りの判定（is_relevantがTrueになるか）になるか確認してください。*

### NGワード設定

「類似度しきい値」や「ネガティブテーマ」では除外しきれない、特定の固有名詞やノイズワードを部分一致で強制除外します。

*   **設定例**: `「地震, 逮捕, 火事, ツールド, 開発行為」`
*   NGワードに一致するキーワードは、NLP判定の前段階でスキップされます。

### Priority Score（優先度スコア）の読み方

Priority Score は「需要の大きさ÷競合の多さ」で決まるブルーオーシャン指標です。

| スコアレンジ | 意味 |
|:---:|:---|
| **高い（目安 > 50）** | 需要はあるのに競合が極めて少ない — 今すぐ書くべきトピック |
| **中程度（10〜50）** | 需要と競合のバランスが取れている — 質で差別化できるトピック |
| **低い（< 10）** | 競合が非常に多い — 独自の切り口なしでは差別化困難 |

> *供給数（supply_count）はQiita APIのタグ別記事数とnoteハッシュタグ投稿数の合算値です。Zennや他プラットフォームの件数は現時点では含まれないため、実際の競合数よりも小さく出る傾向があります。*

## 5. トラブルシューティング

### Q1. `docker compose up` が起動途中でエラーになる、または立ち上がらない
*   **原因①: Dockerデーモンが起動していない**
    *   Docker Desktopが起動しているか確認してください。
*   **原因②: ポートの競合**
    *   すでにホストマシンで `5173` ポート（Vite）や `8000` ポート（FastAPI）が使われている可能性があります。競合しているプロセスを停止するか、`docker-compose.yml` 内の `ports` マッピングを変更してください。

### Q2. データ収集を実行すると「収集失敗」またはタイムアウトする
*   **原因①: Yahoo!の仕様変更によるスクレイピングエラー**
    *   Yahoo!リアルタイム検索のUIが大幅に変更された場合、PlaywrightのXPathによる解析が失敗することがあります。その場合は、`backend/scraper.py` のセレクタ設定を調整する必要があります。
*   **原因②: ネットワーク接続の問題**
    *   初回起動時やモデルのロード時、コンテナ内から外部API（Hugging Face等）へ接続できない場合、一時的に処理がブロックされます。プロキシやインターネット接続状態を確認してください。

### Q3. データ収集しても、すべてのキーワードが「対象外（第3象限）」になってしまう
*   **原因: 類似度しきい値が高すぎる**
    *   「システム設定」で設定した「類似度しきい値」が大きすぎる可能性があります（例: `0.90` など）。しきい値を `0.80` や `0.81` に下げて再試行してください。

### Q4. Brain または Tips のデータが取得できない（0件）
*   **原因①: Cloudflare等によるアクセス拒否**
    *   Brain/Tips はCloudflareのBot保護が強化されることがあります。`playwright-stealth` を適用していますが、保護が強化された場合にはスクレイピングが失敗します。この場合でも他のデータソース（note/Qiita/Zenn）の収集は継続されます。
*   **原因②: URLパス変更**
    *   TipsのランキングURL（現在: `https://tips.jp/rankings`）が変更された場合、`backend/scraper.py` の `fetch_tips_trends()` のURLを更新してください。Brain/Tipsが0件でもシステム全体の動作には影響しません。

### Q5. 供給数（supply_count）がすべて0になる
*   **原因: Qiita APIの応答失敗またはタグ名の不一致**
    *   Qiita APIはタグ名が完全一致（大文字/小文字区別あり）でないと `items_count: 0` を返します。また、ネットワーク問題でAPIが応答しない場合も0になります。供給数が0の場合、Priority Scoreの算出には `log10(0 + 2) ≈ 0.30` が使われるため、スコアが過大評価になることがあります。
