### P1 — 候補キーワードのランダム選択

#### 現状のコード（`scraper.py:560-562`）
```python
candidate_list = list(candidate_keywords)  # Python set → list は順序不定
candidate_list = candidate_list[:40]        # ランダムな40件を採用
```

#### 根本原因
`candidate_keywords` が `set` 型であるため、`list()` 変換後の順序はCPython実装依存（実質的にハッシュ値ベース）。信頼度の高いソース（note人気記事タグ）も低いソース（Brain/Tips）も区別なく混在し、重要なキーワードが毎回取りこぼされる可能性がある。

#### 修正仕様
各収集ソースに**優先度重み**を付与し、重み順に重複排除しながらリストを構築する。同一キーワードが複数ソースで出現した場合は最高優先度を採用。

**ソース優先度定義**:
| ソース | 優先度 | 理由 |
|:---|:---:|:---|
| note人気記事タグ | 5 | 直接の読者実需証拠（スキ数付き） |
| noteシードハッシュタグ関連タグ | 4 | 実需隣接領域の探索 |
| Qiitaトレンドタグ | 3 | テック特化・信頼性高 |
| Zennトレンドキーワード | 3 | テック特化・信頼性高 |
| Yahooサジェスト | 2 | 一般検索由来（ノイズ混在） |
| Brain/Tipsコンテンツキーワード | 2 | スクレイピング不安定性あり |

**実装イメージ（`scraper.py` 内 `run_collection()` を修正）**:
```python
# 現行の candidate_keywords: set を廃止し、優先度付きリストで管理
candidate_map: dict[str, int] = {}  # keyword -> max_priority

def _add_candidate(kw: str, priority: int):
    if len(kw) >= 2 and not kw.isdigit():
        candidate_map[kw] = max(candidate_map.get(kw, 0), priority)

# 各収集ソースの登録時に優先度を渡す
for tag in note_article_tags:   _add_candidate(tag, priority=5)
for tag in note_related_tags:   _add_candidate(tag, priority=4)
for tag in qiita_trends:        _add_candidate(tag, priority=3)
for tag in zenn_trends:         _add_candidate(tag, priority=3)
for tag in suggest_trends:      _add_candidate(tag["keyword"], priority=2)
for tag in brain_trends:        _add_candidate(tag, priority=2)
for tag in tips_trends:         _add_candidate(tag, priority=2)

# 優先度の高い順にソートして上位40件を採用
candidate_list = sorted(candidate_map, key=lambda k: candidate_map[k], reverse=True)[:40]
```

**期待効果**: 毎回同じ収集条件なら同じキーワードが選ばれるようになり、結果の再現性が確保される。

### P2 — 動的ネガティブへの技術ニュース混入

#### 現状のコード（`scraper.py:351-355`）
```python
NEWS_RSS_FEEDS = [
    "https://news.yahoo.co.jp/rss/topics/top-picks.xml",  # ← テックニュースが混入する
    "https://news.yahoo.co.jp/rss/topics/domestic.xml",
    "https://news.yahoo.co.jp/rss/topics/world.xml",
]
```

#### 根本原因
「トップニュース」カテゴリにはAppleの新発表・AIサービス開始などが含まれる。これを動的ネガティブとして使うと、収集当日に話題の正当なテック系キーワードが除外される。

#### 修正仕様
RSSフィードを**確実に非テック**なカテゴリ（スポーツ・エンタメ・社会）に絞る。テック系が混入しやすいトップニュース・国際面は除外する。

```python
NEWS_RSS_FEEDS = [
    "https://news.yahoo.co.jp/rss/topics/sports.xml",        # スポーツ（テック無関係）
    "https://news.yahoo.co.jp/rss/topics/entertainment.xml", # エンタメ（テック無関係）
    "https://news.yahoo.co.jp/rss/topics/domestic.xml",      # 国内社会・政治
]
# 削除: top-picks（テック混入リスク）、world（国際テックニュース混入リスク）
```

**追加対策**: RSSから取得したヘッドラインをそのままネガティブとして使うのではなく、明らかに技術的な単語（「AI」「クラウド」「プログラミング」「エンジニア」「サービス」「リリース」等）を含む見出しを動的ネガティブから除外するプリフィルタを追加する。

```python
TECH_INDICATORS = {"AI", "クラウド", "プログラミング", "エンジニア", "サービス", "リリース",
                   "アプリ", "スマートフォン", "API", "データ", "システム", "DX", "ITシステム"}

def _is_tech_headline(headline: str) -> bool:
    return any(word in headline for word in TECH_INDICATORS)

# fetch_news_headlines() の戻り値フィルタリング
headlines = [h for h in raw_headlines if not _is_tech_headline(h)]
```

### P4 — 4象限境界の毎回変動

#### 現状のコード（`analyzer.py:156-164`）
```python
threshold_x = np.mean(x_positives) / max_x  # 当日データの平均値で変動
threshold_note = np.mean(note_positives) / max_note
threshold_x = max(min(threshold_x, 0.7), 0.3)
threshold_note = max(min(threshold_note, 0.7), 0.3)
```

#### 根本原因
閾値がそのバッチのデータ分布から動的に決まるため、「今週は第1象限だったが来週は第2象限」という分類の揺れが発生する。過去データとの比較・追跡ができない。

#### 修正仕様
`settings` テーブルに `quadrant_mode` と `quadrant_fixed_threshold` を追加し、固定モード（デフォルト）と動的モードを切り替え可能にする。

**DB設定キー追加**（`db.py` の `default_settings`）:
```python
"quadrant_mode": "fixed",          # "fixed" or "dynamic"
"quadrant_fixed_threshold": "0.5", # fixed モード時の閾値（0.0〜1.0）
```

**`analyzer.py` の象限判定を修正**:
```python
settings = local_filter.db.get_settings()
quadrant_mode = settings.get("quadrant_mode", "fixed")
fixed_thr = float(settings.get("quadrant_fixed_threshold", 0.5))

if quadrant_mode == "dynamic":
    # 既存の動的算出ロジック（変更なし）
    threshold_x = ...
    threshold_note = ...
else:
    # 固定モード（デフォルト）
    threshold_x = fixed_thr
    threshold_note = fixed_thr
```

**フロントエンド（Settings.tsx）**: 「象限分類モード」として固定/動的の切り替えUIを追加。固定モード時は閾値スライダーを表示（デフォルト0.5）。

**期待効果**: 固定モードでは過去バッチとの象限比較が可能になり、「先週第4象限 → 今週第1象限」のように成長トレンドを追跡できる。

### P5 — ハッシュタグボーナスの部分一致退化

#### 現状のコード（`analyzer.py:121-122`）
```python
for seed in settings_seeds:
    if seed.lower() in kw.lower():  # 文字列部分一致のみ
```

#### 根本原因
NLPで意味類似度を使う方針と矛盾している。`seed="AI"` の場合、`"AI活用"` はマッチするが `"ChatGPT"` や `"LLM"` はマッチしない。これにより意味的に近いキーワードが不当に低いnote_scoreになる。

#### 修正仕様
`analyze_and_score_trends()` 実行前にシードキーワードの埋め込みベクトルを一括計算し、各候補キーワードとのコサイン類似度でボーナス判定を行う。

```python
# analyzer.py の analyze_and_score_trends() 内で、モデル取得後に追加

# シードキーワードの埋め込みを事前計算（一括処理で高速化）
if settings_seeds:
    seed_embeddings = model.encode(
        [f"passage: {s}" for s in settings_seeds],
        convert_to_tensor=True
    )
else:
    seed_embeddings = None

# 各キーワードのスコア計算ループ内
for item in valid_keywords:
    kw = item["keyword"]
    note_raw_score = 0.0

    # （既存の note_ranking との類似度計算）
    ...

    # ハッシュタグボーナス：NLP類似度ベースに変更
    if seed_embeddings is not None:
        kw_embedding = model.encode(f"query: {kw}", convert_to_tensor=True)
        seed_sims = util.cos_sim(kw_embedding, seed_embeddings)[0]
        for idx, seed in enumerate(settings_seeds):
            sim = float(seed_sims[idx].cpu().item())
            if sim >= 0.80:  # 意味的に近いと判定する閾値
                hashtag_data = db.get_note_hashtag(seed)
                if hashtag_data and hashtag_data["post_count"] > 0:
                    bonus = float(np.log10(hashtag_data["post_count"])) * 2.0 * sim
                    note_raw_score += bonus
```

**計算コスト**: シードが4件の場合、追加の encode コールは `len(valid_keywords)` 回だが、`query:` エンコードは既存処理でも行っているため、`cos_sim` の追加のみで実質的なオーバーヘッドは小さい。

### P3 — supply_count の構造的過小評価

#### 現状の問題
- Qiita API はタグ名完全一致のみ → `"生成AI"` というキーワードは Qiita タグ `"AI"` と一致しないため count=0
- note 側も seed キャッシュにヒットしない限り 0
- 結果: 大多数のキーワードで `supply_count=0` → `Priority Score = demand / log10(2) = demand / 0.30` = 過大評価

#### 修正仕様（2段階）

**段階1（即時対応）: 最小ベースライン供給量の設定**

supply_count が 0 の場合、「データ不足（未調査）」として扱い、Priority Score 計算に中立的なベースライン値を使う。

```python
# analyzer.py の calculate_priority_score() を修正
SUPPLY_BASELINE = 50  # supply_count=0 のとき仮定する競合記事数

def calculate_priority_score(demand_score: float, supply_count: int) -> float:
    if demand_score <= 0:
        return 0.0
    effective_supply = supply_count if supply_count > 0 else SUPPLY_BASELINE
    return demand_score / math.log10(effective_supply + 2)
```

**段階2（追加実装）: Qiita 検索 API による近似供給数取得**

タグ完全一致ではなく、Qiita 検索 API を使ってキーワードを含む記事数を取得する。

```python
async def fetch_qiita_search_count(keyword: str) -> int:
    """Qiita検索APIでキーワードを含む記事の総件数を取得します。"""
    encoded = urllib.parse.quote(keyword)
    url = f"https://qiita.com/api/v2/items?query={encoded}&per_page=1"
    headers = {"User-Agent": random.choice(USER_AGENTS)}
    try:
        async with httpx.AsyncClient() as client:
            response = await client.get(url, headers=headers, timeout=10)
            if response.status_code == 200:
                # Qiita API は X-Total-Count ヘッダーで総件数を返す
                total = int(response.headers.get("X-Total-Count", 0))
                return total
    except Exception as e:
        print(f"Qiita検索カウント取得エラー [{keyword}]: {e}")
    return 0
```

> ⚠️ Qiita API には認証なしで1時間60リクエストのレート制限がある。キーワード数が多い場合は fetch_supply_count() 内でレート制限対策のスリープを追加すること。

**フロントエンドの表示改善**: supply_count=0 かつ SUPPLY_BASELINE を使っているキーワードは Priority Score に `~` プレフィックスを付ける（推定値であることを明示）。

## 4. 修正後のリサーチ方針の整合性

修正後に解消される問題と残存する設計上の留意点を整理する。

### 解消される問題
| 問題 | 修正後の状態 |
|:---|:---|
| 毎回結果が違う（ランダム性） | 優先度ソートにより同条件なら同結果 |
| テックニュースが除外トリガーになる | スポーツ/エンタメ限定RSSで技術混入を排除 |
| 4象限が毎回変わる | 固定閾値モードで過去バッチとの比較が可能 |
| "ChatGPT"がAIボーナスを受けない | NLP類似度ベースで意味的近傍を正しく認識 |
| Priority Score がほぼ無意味 | ベースライン導入で比較可能な相対値に改善 |

### 残存する設計上の留意点（将来の改善候補）

1. **note→noteスコアの自己参照性**: note から収集したキーワードがnoteスコアで評価されるループは構造上残存する。ただし X 軸が独立した評価軸として機能している限り、第1象限（X高・note高）の判定精度には直接影響しない。将来的に「Qiita/Zenn由来のキーワードの比率」を設定で調整可能にすることで緩和できる。

2. **supply_count の精度限界**: Qiita 検索 API でも「記事本文にキーワードが含まれる件数」であり、真の競合（同テーマで書かれた質の高い記事数）とは異なる。Priority Score は絶対値ではなく相対比較のための指標として使用すること。

3. **Yahoo!リアルタイム検索のスクレイピング安定性**: Yahoo の DOM 変更により突然 tweet_count=0 になるリスクは恒常的に存在する。tweet_count=0 のキーワードが多い場合はバッチログで確認し、必要に応じて `scraper.py` のセレクタを更新する。
